Case Study · Custom Moderation Bot

Clicker Evolution

A custom Discord moderation bot written from scratch in Node.js. 25+ commands, tiered auto-punishment, persistent tempbans across restarts, automod with burst-spam detection, and structured logging — built because off-the-shelf bots couldn't be tuned to the server it was made for.

Type Custom bot, single-server
Stack Node.js · discord.js
Commands 25+
Status Status: In production
01

The problem

Off-the-shelf moderation bots — MEE6, Dyno, Carl-bot — get a server to launch quickly, but they impose their own logic on top of yours. Their warn systems don't escalate the way you want them to. Their automod false-positives on the kind of language your community actually uses. Their logs capture a lot of noise and miss what matters. And when one of them goes down, the moderation layer they own goes with it.

Clicker Evolution needed something opinionated about this server's rules, not a generic configuration. Punishment had to escalate gradually but meaningfully. Tempbans had to survive a process restart — losing track of someone's ban timer because the bot rebooted is the kind of bug that erodes member trust. Automod had to detect coordinated burst spam without flagging a normal hyped conversation. None of that was reachable through a settings panel.

02

The solution

A moderation bot written from scratch in Node.js on top of discord.js, with both slash and prefix command support so staff can use whichever interface fits the moment. The architecture splits cleanly into three layers: a command surface (warn, mute, kick, ban, tempban, massban, notes, channel controls), an automation surface (a tiered warn-to-punishment pipeline plus burst-spam and word-filter automod), and a persistence layer that writes moderation state to JSON on disk so nothing — tempban timers, active mutes, warn history — is lost across restarts.

Logging is split by purpose: message edits and deletions land in one channel, moderation actions in another, with DM notifications to the affected member when appropriate. Role-hierarchy protection is enforced at the command layer, so a moderator can't accidentally action someone above them — a common footgun in stock bots.

03

Feature breakdown

WARN SYSTEM
Warns expire 14 days after they're issued. The pipeline auto-escalates as warn count rises — escalation steps run from 3 warns through 15, progressing through mutes, then tempbans, then a permanent ban. Staff don't have to remember the policy; the bot enforces it.
MODERATION COMMANDS
Warn, mute, kick, ban, tempban, massban, notes, and nicknames — each available as both a slash command and a prefix command. Notes are mod-visible only and stay attached to a member's history.
CHANNEL TOOLS
Purge with filters, slowmode, lockdown, and unlock. Channel-scoped so a mod handling a flare-up in one channel never accidentally touches another.
AUTOMOD
Configurable word filter and burst-spam detection (catches coordinated flooding without flagging a normal active conversation). Role-hierarchy protection prevents the bot from actioning members above its own role and stops moderators from being out-ranked into a footgun.
LOGGING
Dedicated channels for message edits and deletions, moderation actions, and member-direction events. DM notifications go to affected members on mutes/bans so they know what happened and why.
PERSISTENCE
JSON-backed storage for warn history, active mutes, and tempban timers. A graceful-shutdown hook flushes state on exit, and on startup the bot rehydrates timers so a tempban that was 4 hours into a 24-hour window before a restart still ends at the right minute.
04

Tech stack

Node.js
Long-running, single-process bots are exactly what Node's event loop is good at — and the deployment story (one process, one host) stays simple enough to debug at 3am.
discord.js
The ecosystem default for Discord bots: tracks Discord API changes quickly, stable enough to depend on, batteries included for slash + prefix command surfaces.
JSON-backed storage
No external database for a single-server bot. JSON on disk + atomic writes + a graceful-shutdown flush is enough state for warns, tempbans, and notes — and easy to inspect when something looks wrong.
Self-hosted
Runs on the client's own infrastructure. Hosting a bot long-term is its own job; keeping it on their box means I'm not on call when their VPS reboots.
05

Outcomes

  • Tempban timers survive process restarts, so members can't out-wait an accidental crash. No timer drift, no “oh the bot rebooted, you're free” edge case.
  • Moderation policy is encoded in code, not staff memory. Escalation from 3 to 15 warns runs the same way at 3am as it does at 3pm.
  • Logging splits noise from signal. Edits and deletions in one channel, mod actions in another — staff stop scrolling past what matters.
  • 25+ commands shipped with full slash + prefix parity. Staff fluent in either interface get the same surface.
